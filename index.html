Shipping a Node.js API without a structured security review is the fastest path to a breach. This guide covers the 30 checks that matter most — the ones that appear repeatedly in real-world vulnerability disclosures and CVE reports — updated for Node 22 LTS and the current Express/Fastify ecosystem.

1. Authentication & Authorization

JWT: Sign with RS256, not HS256

HS256 uses a shared secret — if it leaks from your server, every token ever issued is forged. RS256 uses a keypair: the private key signs, the public key verifies. Your API can verify tokens without knowing the signing secret.

import jwt from 'jsonwebtoken';
import { readFileSync } from 'fs';

const privateKey = readFileSync('./keys/private.pem');
const publicKey  = readFileSync('./keys/public.pem');

// Sign
const token = jwt.sign({ sub: userId, role: 'user' }, privateKey, {
  algorithm: 'RS256',
  expiresIn: '15m',
  issuer: 'api.yourapp.com',
});

// Verify — use publicKey, NOT privateKey
jwt.verify(token, publicKey, { algorithms: ['RS256'] });
Never do this: jwt.verify(token, 'secret') — this allows the alg: none attack and brute-force of weak secrets.

Validate the alg header

Many JWT libraries accept whatever algorithm the token header specifies. Always pin to a specific algorithm. Never use algorithms: ['*'].

Authorization: Check ownership, not just role

The most common API bug is checking that a user is authenticated but not which resources they own. Always verify the resource belongs to the requesting user:

// Wrong — only checks authentication, not ownership
app.get('/api/invoices/:id', requireAuth, async (req, res) => {
  const invoice = await db.invoices.findById(req.params.id);
  res.json(invoice);
});

// Correct — checks ownership
app.get('/api/invoices/:id', requireAuth, async (req, res) => {
  const invoice = await db.invoices.findOne({
    _id: req.params.id,
    userId: req.user.id,  // ← ownership check
  });
  if (!invoice) return res.status(404).json({ error: 'Not found' });
  res.json(invoice);
});

2. Input Validation

Use Zod or Joi at every boundary

Never trust req.body, req.params, or req.query. Validate shape and type before touching your database.

import { z } from 'zod';

const CreateUserSchema = z.object({
  email: z.string().email().max(255),
  name: z.string().min(1).max(100).trim(),
  role: z.enum(['user', 'viewer']),  // never accept 'admin' from client
});

app.post('/api/users', async (req, res) => {
  const result = CreateUserSchema.safeParse(req.body);
  if (!result.success) {
    return res.status(400).json({ errors: result.error.flatten() });
  }
  const user = await createUser(result.data);
  res.status(201).json(user);
});

Prevent NoSQL Injection

MongoDB operators in request bodies can cause privilege escalation. Sanitize before using in queries:

import mongoSanitize from 'express-mongo-sanitize';
app.use(mongoSanitize());  // strips $ and . from keys

SQL Injection: Parameterized queries always

// Wrong
const rows = await db.query(`SELECT * FROM users WHERE id = ${req.params.id}`);

// Correct
const rows = await db.query('SELECT * FROM users WHERE id = $1', [req.params.id]);

3. Security Headers

Add helmet as your first middleware. It sets 15 security headers automatically:

import helmet from 'helmet';
app.use(helmet());
app.use(helmet.contentSecurityPolicy({
  directives: {
    defaultSrc: ["'self'"],
    scriptSrc: ["'self'"],
    objectSrc: ["'none'"],
    upgradeInsecureRequests: [],
  },
}));

4. Rate Limiting

Without rate limiting, your auth endpoints are trivially brute-forced. Add per-IP limits on sensitive routes:

import rateLimit from 'express-rate-limit';

const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  max: 10,
  message: { error: 'Too many attempts' },
  standardHeaders: true,
  legacyHeaders: false,
});

app.post('/api/auth/login', authLimiter, loginHandler);
app.post('/api/auth/forgot-password', authLimiter, forgotPasswordHandler);
IP-based limits aren't enough alone. Attackers rotate IPs. Combine with account-level lockout (e.g., lock after 5 failed attempts for a given email, require CAPTCHA after 3).

5. Secret Management

Never commit secrets to Git

Use environment variables loaded from a secure vault. Add .env to .gitignore immediately. Audit your Git history with git log -p | grep -i secret.

Rotate JWT signing keys on compromise

Implement key versioning with a kid header. Store public keys in a JWKS endpoint. When you rotate, serve both old and new keys for a transition window, then remove the old one.

// JWKS endpoint — let consumers verify without your private key
app.get('/.well-known/jwks.json', (req, res) => {
  res.json({ keys: [getCurrentPublicKeyJWK()] });
});

6. Logging & Monitoring

Log auth failures, permission denials, and rate limit hits. Don't log passwords or tokens — even partially. Route logs to a SIEM (Splunk, Datadog, CloudWatch) with alerting on anomaly patterns.

import pino from 'pino';
const log = pino({ level: 'info', redact: ['req.headers.authorization', 'body.password'] });

app.use((req, res, next) => {
  res.on('finish', () => {
    log.info({ method: req.method, url: req.url, status: res.statusCode, ip: req.ip });
  });
  next();
});

7. Quick Checklist

  • JWT signed with RS256 or EdDSA — never HS256 in production
  • Algorithm pinned in verify call — no wildcard
  • Every protected route checks resource ownership, not just auth
  • Zod/Joi validation on every req.body, req.params, req.query
  • helmet() installed as first middleware
  • Content-Security-Policy configured (not default)
  • Rate limiting on /auth/* and /api/forgot-password
  • .env in .gitignore, secrets in vault not repo
  • SQL via parameterized queries only — no string interpolation
  • express-mongo-sanitize if using MongoDB
  • HTTPS enforced via HSTS header
  • Dependency audit: npm audit fix -- force only for non-breaking
  • Logs redacted — no passwords, no tokens, no PII
  • CORS origin allowlist — not wildcard in production
